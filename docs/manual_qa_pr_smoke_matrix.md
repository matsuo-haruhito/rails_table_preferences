# Manual QA smoke matrix for PRs

Use this matrix when a PR is too small for the full manual QA checklist but still needs a clear browser or review evidence boundary. It complements `manual_qa.md`; it does not replace the full release or host-app launch checklist.

## How to use this matrix

1. Choose the closest PR category below.
2. Run the focused smoke items for that category.
3. Record what was checked, what was skipped, and why in the PR body or a PR comment.
4. Escalate to the full `manual_qa.md` checklist when the PR touches multiple categories, changes runtime UI behavior, changes public helper contracts, or reveals visual/accessibility concerns.

When browser capture is not available, use the strongest available substitute: focused system spec, DOM assertion, static visual reference check, source-level invariant, or a clear `needs-human` handoff. Do not describe source inspection as visual evidence when the acceptance criteria require rendered UI proof.

## PR category matrix

| Docs-only | Confirm links, commands, option names, and issue/PR references match current source. When README, docs index, English quick start, or focused guide entrypoints change, compare `docs/quick_start_ja.md` links and short summaries for low-drift alignment without requiring a full translation. No browser smoke is required unless the doc claims a rendered state. | Docs paths checked, source-of-truth files consulted, and any intentionally skipped browser checks. For Japanese quick start drift checks, record which README/docs index/focused English docs entrypoints were compared and whether only links/short summaries changed. | The doc changes visible UI promises, public API guarantees, release sign-off wording, or asks for a full bilingual docs rewrite instead of a low-drift entrypoint check. |
| Static visual docs or docs image | Confirm the changed SVG, screenshot, or docs image still matches the surrounding caption and source-of-truth docs. If the visual file changed, render or open the artifact when available; otherwise record source inspection and leave browser-capable review for the rendered view. | Changed visual/doc paths, source-of-truth docs consulted, whether rendered visual confirmation or source-only inspection was used, and any browser-capable handoff. | The PR claims production UI behavior, changes a runtime surface, requires screenshot approval, or the rendered visual cannot be checked despite acceptance criteria requiring it. |
| Turbo Drive or Turbo Frame reconnect | After Turbo navigation or frame replacement, confirm the editor and matching table still point at the same logical screen: `data-controller="rails-table-preferences"`, the same table key, current settings and columns values, collection/member URL values, and matching managed header/cell column keys. Prefer replacing the editor and table together, or passing the same settings and columns to both pieces. | Screen or frame checked, whether the editor/table were replaced together, DOM/root values or rendered source snippets confirmed, and links to the [Turbo checks](javascript_entrypoints.md#turbo-drive-and-turbo-frame-checks) and [table data attribute boundary](table_data_attributes.md) when frame data is involved. | The PR needs host-app Turbo routing, Turbo Stream response, or query execution changes; the editor and table render from different responses; or the reconnected DOM is incomplete or inconsistent. |
| Helper-free table or manual controller root | For host-owned table markup that bypasses `table_preferences_table_tag(...)`, inspect the rendered controller root and table. Confirm `table_key`, `name` when an editor or preset select is present, `columns`, `settings`, collection/member URL values, and managed header/cell column keys all describe the same logical table screen. | Rendered DOM snippet, source-level assertion, or browser notes showing the manual root values, editor/preset partial used if any, collection/member URL values, and representative managed column keys. Note whether the same values are passed into separate table and editor partials. | The PR adds a helper API, changes generated table markup, introduces custom partial contracts, changes Turbo split-render behavior, or cannot show that the editor and table share one root-value contract. |
| Controller or bundled editor UI | Open a representative editor screen, apply one visibility/order change, and check desktop plus one narrow width for clipping, focus visibility, and helper/status copy. When action grouping is touched or used as evidence, confirm `適用` / save / save-as-new / delete / reset controls still wrap as separate primary, save, and maintenance groups; long Japanese labels or hints must not merge maintenance actions into the Apply/Save area. When editor row layout, dense labels, or narrow-row evidence is touched, use the focused [editor row long-label narrow smoke](editor_row_long_label_narrow_smoke.md) note for representative labels, widths, and handoff wording. For filter panel changes, use a short viewport with a multi-select or several-field panel and confirm Apply / Clear remain reachable. | Screenshot, browser notes, focused system spec, or DOM evidence for desktop and narrow width. For action grouping, record the viewport, label length or locale checked, whether forced-colors/high-contrast was checked or handed off, and whether primary/save/maintenance groups stayed distinguishable and reachable. For editor row labels, record the long Japanese or unbroken label examples, the 320px / 375px / 390px-equivalent or narrow container checked, and whether the focused row smoke was rendered or handed off. For filter panels, record the viewport height, representative field type, and how Apply / Clear were reached. | The PR changes async preset actions, drag/resize/filter/sort interactions, accepted keyboard behavior, action-group styling, editor row wrapping or dense-label layout, filter panel positioning, filter panel action reachability, or claims forced-colors/high-contrast evidence without rendered confirmation. |
| Column overflow metadata | For `overflow:` / `default_overflow:` docs, demo, CSS, or visual-reference changes, compare representative `:ellipsis`, `:clip`, `:wrap`, and `:nowrap` states in the generated demo or one real host-app table. Keep this as rendered display evidence for column-definition metadata, not as evidence that the bundled editor has an overflow selector. | Screen checked, representative column, viewport or container width, overflow modes exercised, whether demo or host-app table evidence was used, and any browser-capable handoff when only source/docs were inspected. | The PR changes runtime overflow styling, generated demo examples, saved settings shape, editor controls, truncation precedence, or claims visual proof without rendered evidence. |
| Package lifecycle event listener | Register a representative listener on a screen that uses the package entrypoint, then perform one operation the PR claims: apply, reset, save, load, delete, or an error-path operation. Confirm the generated copied controller path is not treated as event-capable unless the host app ports the same behavior. | Event name, package entrypoint registration path, operation performed, observed `event.detail` fields such as `tableKey`, `name`, `action`, `settings`, or error `message`, and any copied-controller handoff. | The PR changes event names, detail payload shape, copied-controller parity, reset event semantics, analytics provider integration, or requires browser automation beyond a representative listener smoke. |
| Large select filter option search | On a representative select filter with enough options to show the option-search input, open the filter panel in a short viewport and confirm keyboard focus can move from the search input to the multi-select and then to Apply/Clear through panel scrolling. Search for text that hides most unselected options and confirm already-selected options remain visible enough to review before applying; keep automated source-level selected-option guards separate from this manual/browser smoke. | Browser notes or screenshot naming the representative column, viewport height or container, option count or threshold setup, keyboard path used, whether selected options stayed visible during search, and how Apply/Clear were reached. | The PR changes option-search filtering behavior, selected option preservation, filter panel positioning, remote/async option loading, external select widgets, query execution, or claims rendered keyboard/scroll evidence without a browser-capable check. |
| Resource table helper or view partial | Render records-present and records-empty states, including all-hidden columns when relevant. Confirm captions, row/cell hooks, and `colspan` stay valid. When a resource table profile formatter reads associations, confirm the host app relation preloads those associations and that a representative query log or existing N+1 guard stays clean, matching the full manual QA formatter check. For association-reading formatter changes, record the formatter/profile checked, the preload path or relation owner, and the query-log or N+1 guard evidence used. For custom editor placement, record the placement checked, matching table key/settings/columns, apply/save/load smoke, narrow or short viewport result, and any host-app-owned drawer/tab responsibility. | The PR adds a public data hook, new helper option, custom partial contract, or tree-table behavior. Escalate custom editor placement when the editor and table are rendered from mismatched responses, drawer/tab focus management is undecided, or the screen needs a new placement API. |
| Generator or demo flow | Run or inspect the generator path touched by the PR, then confirm generated file, route, setup note, and demo screen guidance match. When the generated demo surface is in scope, use [Demo screen generator](demo.md) as the detailed source of truth and include or explicitly hand off the relevant support sections: `Current owner` summary and owner switching links, optional `Current scope context` and role/organization lanes, `Demo state reset`, one-shot `Async failure check`, hidden fields preview, and export payload preview. Treat source inspection, rendered demo evidence, and browser-capable handoff as separate evidence types. | Generator command/path, files expected to change, whether route/setup idempotency was checked, whether owner/scope summaries or links were checked or handed off, whether reset/failure support sections were rendered or handed off, whether hidden-fields/export previews were checked or handed off, and why any browser check was skipped. | The PR changes install defaults, migration shape, mounted route behavior, post-install instructions, demo owner/scope links, demo reset behavior, async failure helper behavior, hidden-fields/export preview behavior, or claims rendered demo evidence that was not actually checked. |
| Export, hidden fields, or controller params | Exercise one representative saved preference state and confirm the outgoing payload/hidden fields/params match current docs. For `include_hidden: true` export flows, confirm hidden columns are treated as display preferences and record the host app allowlist, serializer, or policy evidence that decides which hidden or sensitive columns may be exported. | Focused spec or payload sample showing included, omitted, and mapped params. For `include_hidden: true`, add the export policy or serializer evidence used and note whether the flow is available to all users, admin-only users, or a narrower export path. | The PR changes adapter precedence, search execution assumptions, export column selection, host-app ownership boundaries, or asks Rails Table Preferences to decide sensitive-column authorization. |
| Fixed columns, grouped headers, or dense layout | Check one horizontally scrolled table, one focusable cell control, and one narrow or long-label case. Keep the current gem surface scoped to left-pinned hooks; if the PR discusses right-pinned columns or complex sticky engines, record that boundary as deferred/host-owned rather than claiming rendered support. | Browser notes or screenshot showing left-pinned/fixed overlap, focus visibility, grouped header alignment, and whether any right-pinned or complex sticky behavior was intentionally out of scope. | The PR changes sticky offsets, z-index guidance, grouped header markup, scroll-container assumptions, or implies right-pinned / multi-scroll-container support beyond the current non-goals boundary. |
| UI, CSS, accessibility, or high-contrast state | Use [Accessibility baseline](accessibility.md#forced-colors-and-high-contrast-checks) as the source of truth and check the representative states touched by the PR: active filter buttons, sorted headers, focused resize handles, filter panel boundaries, editor action groups, and pinned/fixed columns. Treat rendered forced-colors / high contrast evidence as the primary proof for visual claims. Source-level or DOM guards can support invariants such as `aria-sort`, `aria-pressed`, disabled state, labelled controls, or CSS hook presence, but they do not prove contrast, overlap, focus visibility, or stacking. For tall filter panels, pair this with the Controller or bundled editor UI row so short-viewport action reachability is recorded separately from color/focus checks. | Browser notes, screenshot, or browser-capable handoff for the forced-colors / high contrast mode used, representative states checked, skipped states and reason, any source-level invariants used only as supporting evidence, and any host-app theme responsibility that remains outside the gem. | The PR changes runtime CSS, color tokens, focus styles, filter panel layering, editor action grouping, pinned/fixed column styling, or claims rendered accessibility evidence that cannot be checked in the current environment. |
| Scoped presets | Confirm owner, shared, role, or organization options remain distinguishable and that read-only presets do not expose destructive actions as editable. | Selector option labels/scope notes, owner/context used, and any read-only hint checked. | The PR changes default resolution, write scope, authorization expectations, or selector grouping behavior. |
| Downstream host-app bump or adoption smoke | For a downstream app PR, pick one representative real table surface and use the [downstream adoption evidence template](production_integration_checklist.md#downstream-adoption-evidence-template) as the source of truth. Keep the upstream gem PR evidence separate from host-app-specific routes, schemas, release gates, and rollback targets. | Host app/surface name, gem version or SHA, stable table and column keys, filter/sort mapping, preset save/load path, mounted engine auth boundary, export boundary if used, layout/accessibility smoke, rollback target, and human release owner. | The downstream known-good target, rollback target, or release owner is undecided; the host-app bump needs route/schema changes; or the PR is trying to turn host-app-specific behavior into upstream Rails Table Preferences contract. |

## Review-state handoff checks

Use this section when the PR is blocked by review state rather than by another browser smoke item.

| Review state | What to check | Evidence to record | Escalate when |
| --- | --- | --- | --- |
| Public contract review | Confirm whether the PR changes a user-facing helper option, package entrypoint, event payload, release sign-off wording, or package verification boundary. Keep browser evidence separate from the maintainer decision about whether the contract should exist. | Source-of-truth issue or PR, current code/spec/docs checked, why human review remains, and any behavior that is intentionally not being claimed yet. | The PR needs a new public API decision, verifier required-path decision, release policy decision, or compatibility promise that docs cannot decide alone. |
| Stacked dependency | Confirm the PR base branch, parent PR, and whether the text should use `Refs` instead of `Closes` until the parent behavior reaches `main`. Do not document unmerged parent behavior as current `main` behavior. If the parent PR was closed or superseded by a replacement, record the replacement PR and whether the child should wait, move to the replacement parent, or be rebuilt from current `main`. | Base branch, parent PR, dependency status, `Refs` / `Closes` choice, replacement PR when one exists, and what should be rechecked after the parent or replacement merges. | The parent PR is unmerged, closed without landing, superseded with uncertain scope, mergeability is uncertain, or the docs would need to move to `main` before the behavior exists there. |
| Replacement / superseded PR | Confirm the replacement PR names the superseded PR, carries the needed diff, and is ready to be the review target before the old PR is treated as a close candidate. Check replacement draft state, CI or failure summary, mergeability, review state, and changed files. | Replacement PR, superseded PR, head SHA, compare or changed-file evidence, CI/review state, old-PR handoff comment, and whether the old PR is only a close candidate or already closed. | The replacement has not passed or summarized CI, the old PR may still contain needed scope, review comments require human judgment, or closing the old PR would hide unresolved work. |
| Mergeability refresh | Check `behind_by`, `mergeable`, changed files, and whether the branch can be refreshed or should be replaced from current `main`. CI success alone is not enough when GitHub reports `mergeable:false` or the compare is diverged. | Compare state, head SHA, CI result, changed files, and whether the next action is branch refresh, clean replacement, or human handoff. | Refreshing would pull unrelated scope into the PR, conflict resolution is needed, or deciding which side wins requires implementation or product judgment. |
| Browser evidence handoff | Confirm whether the acceptance criteria require rendered UI proof, screenshot review, forced-colors/high-contrast mode, or a real host-app screen. Source inspection may support the handoff, but it is not rendered evidence. | Required viewport/state, environment that could not be checked, strongest substitute used, and the exact browser-capable check left for review. | The PR claims visual/accessibility results without rendered evidence, or the remaining check needs a real app, real data, or browser access unavailable to the current agent. |

## PR comment template

```markdown
### Focused manual QA

- PR category:
- Review state or handoff reason:
- Screen or artifact checked:
- Viewports or states checked:
- Evidence:
- Skipped full-checklist areas and reason:
- Follow-up needed:
```

Keep the note short. For docs-only and spec-only changes, it is fine to say browser QA was not applicable, as long as the changed text does not claim a rendered UI state. For UI and visual reference changes, include rendered evidence or leave a `needs-human` handoff for someone with a browser-capable environment.
